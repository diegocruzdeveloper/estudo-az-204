Guia Completo para a Certificação Microsoft Azure AZ-204

Desenvolvendo Soluções para Microsoft Azure

Este guia detalha os principais tópicos e subitens cobrados na prova de certificação AZ-204, organizada pelas áreas de conhecimento, para ajudar na sua preparação.

Desenvolver soluções de computação do Azure (25–30%)

Esta seção abrange os conhecimentos e habilidades necessários para criar e gerenciar diferentes tipos de soluções de computação na nuvem Azure.

1. Implementar soluções conteinerizadas

Este tópico é fundamental para quem busca a certificação AZ-204 e abrange a criação, publicação e execução de contêineres no Azure.

1.1. Criar e gerenciar imagens de contêiner para soluções

Para a certificação AZ-204, você precisa ser proficiente em:

  • Dockerfiles:
    • Estrutura e Sintaxe: Entender a estrutura de um Dockerfile e as instruções mais comuns, como FROM, RUN, COPY, ADD, WORKDIR, EXPOSE, ENV, CMD e ENTRYPOINT.
    • Melhores Práticas: Saber como criar Dockerfiles eficientes e seguros, incluindo o uso de imagens base menores, camadas otimizadas, multi-stage builds e a minimização de dependências desnecessárias.
    • Variáveis de Ambiente e Argumentos de Build: Utilizar ENV para variáveis de ambiente em tempo de execução e ARG para variáveis em tempo de build.
  • Construção de Imagens:
    • docker build: Dominar o comando docker build para criar imagens a partir de um Dockerfile, incluindo o uso do contexto (.) e a especificação de tags (-t).
    • Cache de Build: Entender como o Docker utiliza o cache para acelerar o processo de build e como invalidá-lo quando necessário.
  • Gerenciamento Local de Imagens:
    • Listar Imagens: Usar docker images ou docker image ls para visualizar as imagens disponíveis localmente.
    • Remover Imagens: Empregar docker rmi para remover imagens indesejadas.
    • Inspeção de Imagens: Utilizar docker inspect para obter detalhes sobre uma imagem.
    • Salvar e Carregar Imagens: Conhecer docker save e docker load para exportar e importar imagens como arquivos.

1.2. Publicar uma imagem no Registro de Contêiner do Azure

Este tópico é crucial para disponibilizar suas imagens para implantação no Azure:

  • Azure Container Registry (ACR):
    • Criação e Configuração: Saber como criar uma instância do ACR no portal do Azure, via Azure CLI (az acr create) ou PowerShell.
    • SKUs do ACR: Entender as diferentes SKUs (Basic, Standard, Premium) e suas características (armazenamento, largura de banda, replicação geográfica).
    • Segurança: Configurar redes virtuais, pontos de extremidade privados e regras de firewall para o ACR.
  • Autenticação no ACR:
    • Login do Docker: Usar az acr login --name <nome_do_acr> para autenticar seu cliente Docker localmente.
    • Entidades de Serviço: Criar e configurar entidades de serviço com as permissões apropriadas para acessar o ACR de pipelines de CI/CD ou outras aplicações.
    • Chaves de Administrador: Entender (e desaconselhar para produção) o uso de chaves de administrador para acesso.
  • Tagging e Push de Imagens:
    • Tagging: Renomear suas imagens locais com o nome de login do ACR (docker tag <imagem_local> <login_server_acr>/<repositorio>:<tag>).
      • Ex: docker tag myapp:v1 myregistry.azurecr.io/myapp:v1.
    • Push: Enviar a imagem "taggeada" para o ACR usando docker push <login_server_acr>/<repositorio>:<tag>.
  • Tarefas do ACR (ACR Tasks):
    • Builds Automatizados: Utilizar ACR Tasks para automatizar a construção de imagens Docker a partir de código-fonte (GitHub, Azure Repos) ou atualizações de imagem base.
    • Multi-step Tasks: Criar tarefas mais complexas que envolvem múltiplos passos, como build, test e push.

1.3. Executar contêineres usando a Instância de Contêiner do Azure

O ACI é ideal para cenários de execução rápida e simples de contêineres:

  • Conceito e Casos de Uso:
    • Contêineres Sem Servidor: Entender que o ACI permite executar contêineres sem a necessidade de provisionar ou gerenciar máquinas virtuais subjacentes.
    • Cargas de Trabalho de Estouro: Cenários onde você precisa de capacidade adicional rapidamente.
    • Tarefas Orientadas a Eventos: Execução de tarefas de processamento de dados, automação ou scripts.
    • Ambientes de Desenvolvimento/Teste: Implantação rápida para testes ou demonstrações.
  • Implantação de Contêineres:
    • Azure CLI: Dominar o comando az container create, especificando:
      • Nome da imagem (--image)
      • Nome do grupo de contêineres (--name)
      • Recursos (CPU e memória: --cpu, --memory)
      • Portas expostas (--ports)
      • Variáveis de Ambiente (--environment-variables)
      • Comandos de Inicialização (--command-line)
      • Montagem de Volumes (Azure File Share: --azure-file-volume-account-name, --azure-file-volume-share-name, --azure-file-volume-mount-path)
    • YAML: Criar arquivos YAML para definir grupos de contêineres mais complexos.
  • Gerenciamento e Monitoramento:
    • Iniciar/Parar/Reiniciar: Usar az container start, az container stop, az container restart.
    • Logs: Visualizar logs de contêineres com az container logs.
    • Executar Comandos: Conectar-se a um contêiner em execução para depuração com az container exec.
    • Monitoramento: Integrar com o Azure Monitor para métricas e logs.

1.4. Criar soluções usando os Aplicativos de Contêiner do Azure (e orquestradores/webhooks)

Os Aplicativos de Contêiner do Azure (Azure Container Apps) são uma plataforma mais recente e poderosa para microsserviços e aplicações conteinerizadas. Orquestradores e Webhooks se encaixam aqui como conceitos complementares.

  • Conceito e Benefícios dos Aplicativos de Contêiner do Azure:
    • Plataforma para Microsserviços: Projetado para hospedar microsserviços e aplicações baseadas em contêineres, incluindo aqueles que usam Dapr e KEDA.
    • Serviço Totalmente Gerenciado: Abstrai a complexidade do Kubernetes, permitindo que você se concentre no código.
    • Dimensionamento Baseado em Eventos: Suporte nativo para KEDA (Kubernetes Event-Driven Autoscaling), permitindo dimensionamento baseado em filas de mensagens, tópicos Kafka, HTTP e muito mais.
    • Dapr (Distributed Application Runtime): Integração com Dapr para simplificar a construção de microsserviços (gerenciamento de estado, pub/sub, service invocation, etc.).
    • Revisões: Suporte a revisões para implantações sem tempo de inatividade e rollbacks.
  • Componentes Chave dos Aplicativos de Contêiner:
    • Ambientes de Aplicativos de Contêiner: Onde seus aplicativos de contêiner são implantados.
    • Aplicativos de Contêiner: A unidade de implantação, que contém uma ou mais revisões.
    • Revisões: Versões imutáveis de um aplicativo de contêiner.
    • Dimensionamento: Configuração de regras de dimensionamento (HTTP, CPU, KEDA scalers).
    • Entrada (Ingress): Configuração de como o tráfego externo acessa seu aplicativo de contêiner.
  • Implantação e Gerenciamento:
    • Azure CLI: Usar az containerapp create para criar aplicativos de contêiner, especificando a imagem, recursos, variáveis de ambiente e regras de dimensionamento.
    • Dapr e KEDA: Configurar Dapr para funcionalidades de microsserviços e KEDA para dimensionamento baseado em eventos.
    • Gerenciamento de Revisões: Gerenciar diferentes revisões de um aplicativo de contêiner para implantações canary ou blue/green.
    • Monitoramento: Integrar com Azure Monitor e Application Insights.
  • Orquestradores (Contexto AZ-204):
    • Kubernetes (AKS): O principal orquestrador que você precisa entender para o AZ-204. Ele é usado para automatizar a implantação, dimensionamento e gerenciamento de aplicações conteinerizadas.
    • Conceitos Fundamentais: Pods, Deployments, Services, Namespaces, Ingress, ConfigMaps, Secrets, Persistent Volumes.
    • Implantação de Aplicações: Uso de arquivos YAML e kubectl para definir e gerenciar recursos do Kubernetes.
    • Dimensionamento: Horizontal Pod Autoscaler (HPA) e Cluster Autoscaler.
    • Atualizações Rollout/Rollback: Estratégias de atualização de aplicações.
    • Azure Container Apps: Embora seja um serviço mais gerenciado, ele internamente utiliza conceitos de orquestração (como KEDA para dimensionamento) que se assemelham a orquestradores de contêineres.
  • Webhooks (Contexto AZ-204):
    • Automação e Integração: Webhooks são usados para permitir que um serviço notifique outro serviço sobre um evento. No contexto de contêineres, eles são frequentemente usados em cenários de CI/CD e automação.
    • ACR Tasks: As ACR Tasks podem ser configuradas para disparar builds de imagem automaticamente quando o código-fonte é atualizado em um repositório Git (usando webhooks do GitHub/Azure Repos).
    • Azure DevOps/GitHub Actions: Pipelines de CI/CD em Azure DevOps ou GitHub Actions podem ser configurados para serem disparados por webhooks de repositórios de código ou de registros de contêiner (ex: quando uma nova imagem é enviada para o ACR).
    • Azure Functions/Logic Apps: Webhooks podem ser o gatilho para Azure Functions ou Logic Apps que executam ações em resposta a eventos relacionados a contêineres (ex: notificar uma equipe quando um contêiner falha, ou disparar um processo de auditoria quando uma nova imagem é implantada).
    • Integração com Ferramentas Externas: Webhooks permitem que suas soluções conteinerizadas se integrem com sistemas de monitoramento, ferramentas de notificação (Slack, Teams) ou outros serviços externos.

2. Implementar Aplicativos Web do Serviço de Aplicativo do Azure

Este tópico se concentra nos recursos do Azure App Service para hospedar e gerenciar aplicações web.

2.1. Criar um Aplicativo de API do Serviço de Aplicativo do Azure

  • Conceito de Aplicativo de API: Entender que um Aplicativo de API no App Service é otimizado para hospedar APIs RESTful, fornecendo recursos como CORS (Cross-Origin Resource Sharing) e integração com o Azure API Management.
  • Criação via Portal/CLI/PowerShell: Saber como provisionar um Aplicativo de API, escolhendo o runtime (Node.js, .NET, Java, Python, etc.) e o plano do Serviço de Aplicativo.
  • Publicação de API: Entender os métodos de publicação, como o uso de ferramentas de publicação do Visual Studio, FTP, Git local ou integração com CI/CD.

2.2. Configurar e implementar diagnósticos e registros

A capacidade de monitorar e depurar suas aplicações é crucial:

  • Logs do Aplicativo:
    • Tipos de Log: Entender os diferentes tipos de logs disponíveis (logs de aplicativo, logs do servidor web, mensagens de erro detalhadas, rastreamento de solicitações com falha).
    • Níveis de Log: Configurar o nível de detalhe dos logs (erro, aviso, informação, depuração).
    • Armazenamento: Habilitar o envio de logs para Armazenamento de Blobs do Azure, Log Analytics ou stream de logs.
  • Logs do Servidor Web: Ativar e analisar logs de requisições HTTP para depurar problemas de conectividade ou desempenho.
  • Azure Application Insights:
    • Configuração: Integrar o Application Insights à sua aplicação para monitoramento de desempenho de aplicativos (APM).
    • Recursos: Utilizar mapas de aplicativo, gráficos de desempenho, telemetria de requisições e exceções.
  • Diagnóstico e Solução de Problemas: Utilizar as ferramentas de diagnóstico integradas no portal do Azure para identificar problemas comuns.

2.3. Implantar código e soluções em contêineres

Este subitem aborda os diferentes métodos de implantação:

  • Implantação de Código (Não Contêiner):
    • Git Local/GitHub/Azure Repos: Configurar a implantação contínua a partir de repositórios de código.
    • ZIP Deploy: Utilizar o comando az webapp deployment source config-zip para implantar pacotes ZIP.
    • FTP/Web Deploy: Métodos tradicionais de implantação.
  • Implantação de Soluções em Contêineres:
    • Imagens Docker: Implantar imagens do Docker Hub, Azure Container Registry (ACR) ou registros de contêiner privados.
    • Configuração de Contêineres: Definir a imagem, porta do contêiner, variáveis de ambiente e comandos de inicialização para o contêiner.
    • Implantação Contínua com Contêineres: Configurar webhooks ou pipelines de CI/CD para que o App Service puxe automaticamente a imagem mais recente do ACR quando ela for atualizada.
    • Docker Compose (para multi-contêiner): Entender como usar arquivos Docker Compose para implantar aplicações com múltiplos contêineres no App Service.

2.4. Definir configurações, incluindo o protocolo TLS (TLS), configurações de API e conexões de serviço

A configuração adequada é vital para a segurança e funcionalidade:

  • Certificados TLS/SSL:
    • Gerenciamento: Entender como vincular domínios personalizados e carregar certificados SSL (incluindo certificados Let's Encrypt via extensões do App Service).
    • Imposição HTTPS: Configurar o aplicativo para redirecionar todo o tráfego HTTP para HTTPS.
  • Variáveis de Ambiente e Configurações de Aplicativo:
    • Configurações de Aplicativo: Gerenciar pares chave-valor para configurações da aplicação (string de conexão, chaves de API) que podem ser injetadas como variáveis de ambiente no runtime.
    • Prioridade: Entender a precedência entre variáveis de ambiente do sistema operacional, configurações do aplicativo e configurações de conexão.
  • Cadeias de Conexão: Configurar cadeias de conexão para bancos de dados e outros serviços, com segurança aprimorada.
  • CORS (Cross-Origin Resource Sharing): Configurar regras CORS para permitir que clientes de diferentes origens acessem sua API de forma segura.
  • Identidades Gerenciadas:
    • Conceito: Utilizar identidades gerenciadas (atribuídas ao sistema ou atribuídas pelo usuário) para permitir que seu aplicativo acesse outros serviços do Azure (Key Vault, SQL Database) de forma segura, sem a necessidade de credenciais embutidas.
    • Implementação: Habilitar e configurar permissões para identidades gerenciadas.

2.5. Implementar o dimensionamento automático

Garantir que sua aplicação possa lidar com diferentes volumes de tráfego:

  • Tipos de Dimensionamento:
    • Escala Manual: Ajustar o número de instâncias manualmente.
    • Dimensionamento Automático (Autoscale): Configurar regras para que o App Service dimensione o número de instâncias com base em métricas.
  • Regras de Dimensionamento:
    • Métricas: Basear regras em métricas como CPU, memória, fila de HTTP, dados de saída, ou filas de armazenamento/Service Bus.
    • Limites: Definir limites mínimos e máximos de instâncias.
    • Períodos de Tempo: Configurar horários específicos para regras de dimensionamento.
    • Múltiplas Regras: Criar múltiplas regras para diferentes cenários de carga.
  • Planos do Serviço de Aplicativo: Entender como os planos de serviço afetam as opções e capacidades de dimensionamento.

2.6. Configurar slots de implantação

Essencial para implantações sem tempo de inatividade e testes:

  • Conceito de Slots de Implantação:
    • Ambientes de Staging: Entender que os slots são instâncias separadas de sua aplicação que compartilham o mesmo plano do Serviço de Aplicativo, mas com URLs e configurações independentes.
    • Trocas (Swapping): Compreender o processo de troca de slots (troca ao vivo ou troca com visualização) para mover uma versão testada de um slot de preparo para o slot de produção sem tempo de inatividade.
  • Criação e Gerenciamento:
    • Adicionar Slots: Criar novos slots de implantação no portal do Azure ou via CLI (az webapp deployment slot create).
    • Configurações Específicas do Slot: Saber como configurar variáveis de ambiente, cadeias de conexão e outras configurações que são específicas de um slot e não são trocadas com o slot de produção.
  • Cenários de Uso:
    • Implantações Blue/Green: Usar slots para testar novas versões em um ambiente de produção antes de redirecionar o tráfego.
    • Testes A/B: Direcionar uma pequena porcentagem do tráfego para um novo slot para testes.
    • Rollback Rápido: Em caso de problema, reverter rapidamente para uma versão anterior trocando de volta para um slot estável.

3. Implementar Funções do Azure

Este tópico é crucial para entender como criar e gerenciar soluções serverless com o Azure Functions.

3.1. Criar e configurar um aplicativo do Azure Functions

Para criar e gerenciar um Function App, você precisa dominar:

  • Conceito de Azure Functions: Entender o Azure Functions como uma solução de computação serverless baseada em eventos, ideal para executar pequenos trechos de código (funções) sem a necessidade de gerenciar infraestrutura.
  • Planos de Hospedagem:
    • Plano de Consumo: Saber que este plano é serverless puro, escalando automaticamente de acordo com a carga e cobrando apenas pelo tempo de execução e consumo de memória. É o plano mais comum para funções baseadas em eventos e uso intermitente.
    • Plano Premium (Elastic Premium): Conhecer este plano para cenários que exigem baixa latência, conectividade VNet e pré-aquecimento de instâncias, com um custo base.
    • Plano do Serviço de Aplicativo (Dedicated Plan): Entender que suas funções podem ser executadas em um plano de Serviço de Aplicativo existente para ter maior controle sobre os recursos e um custo fixo, mas sem os benefícios serverless de dimensionamento automático e cobrança por execução.
  • Criação do Function App:
    • Portal do Azure/Azure CLI/PowerShell: Saber como criar um Function App, especificando o nome, runtime (C#, JavaScript, Python, Java, PowerShell), região, plano de hospedagem e conta de armazenamento.
    • Ferramentas de Desenvolvimento: Usar Visual Studio, Visual Studio Code ou Azure Functions Core Tools para criar e depurar funções localmente.
  • Configurações do Aplicativo de Função:
    • Variáveis de Ambiente: Gerenciar variáveis de ambiente para configurações da função, como strings de conexão e chaves de API.
    • Configurações de Conexão: Configurar conexões para outros serviços do Azure (ex: Service Bus, Cosmos DB) nas configurações do Function App.
    • Identidades Gerenciadas: Habilitar e configurar identidades gerenciadas para permitir que suas funções acessem outros serviços do Azure com segurança, sem credenciais embutidas.

3.2. Implementar associações de entrada e de saída

As associações são uma das características mais poderosas do Azure Functions, simplificando a integração de dados:

  • Conceito de Associações (Bindings): Entender que as associações fornecem uma maneira declarativa de conectar suas funções a dados e serviços. Elas eliminam a necessidade de escrever código complexo para interagir com esses serviços.
  • Tipos de Associações: Familiarizar-se com associações comuns:
    • Associações de Entrada (Input Bindings): Permitem que sua função leia dados de um serviço. Ex: ler uma mensagem de uma fila do Azure Storage, obter um documento do Cosmos DB, recuperar um blob do Azure Blob Storage.
    • Associações de Saída (Output Bindings): Permitem que sua função grave dados em um serviço. Ex: enviar uma mensagem para uma fila ou tópico do Service Bus, gravar um documento no Cosmos DB, criar um blob no Azure Blob Storage.
  • Configuração de Associações:
    • function.json (para JavaScript/PowerShell/Python): Entender como definir as associações no arquivo function.json, especificando o tipo de ligação, nome do parâmetro e detalhes da conexão.
    • Atributos C#: Usar atributos específicos (ex: [BlobInput], [QueueOutput], [CosmosDB]) para definir associações diretamente no código C#.
  • Utilização de Associações no Código: Saber como os dados das associações são passados para sua função como parâmetros de entrada e como retornar dados ou usar parâmetros out para associações de saída.

3.3. Implementar gatilhos função usando operações de dados, temporizadores e webhooks

Gatilhos são o que fazem sua função ser executada:

  • Conceito de Gatilho (Trigger): Entender que um gatilho é um evento que inicia a execução de uma função. Cada função tem exatamente um gatilho.
  • Tipos de Gatilhos: Dominar os gatilhos mais comuns e seus casos de uso:
    • Gatilho de Operações de Dados:
      • Azure Blob Storage Trigger: Executa a função quando um novo blob é adicionado ou um blob existente é modificado. Útil para processamento de arquivos.
      • Azure Queue Storage Trigger: Executa a função quando uma nova mensagem é adicionada a uma fila. Perfeito para processamento de mensagens assíncronas.
      • Azure Cosmos DB Trigger: Executa a função em resposta a mudanças em documentos em uma coleção do Cosmos DB (via Change Feed). Ideal para cenários de processamento em tempo real de dados.
      • Azure Service Bus Trigger: Executa a função quando uma nova mensagem é adicionada a uma fila ou tópico do Service Bus. Permite comunicação desacoplada e padrões de mensagens avançados.
      • Event Hubs Trigger: Executa a função quando novos eventos são adicionados a um event hub. Usado para ingestão de dados em larga escala e processamento de streams.
    • Gatilho de Temporizador (Timer Trigger):
      • Expressões CRON: Saber como usar expressões CRON para agendar a execução de uma função em intervalos regulares (ex: a cada hora, diariamente à meia-noite).
      • Casos de Uso: Tarefas agendadas, manutenção de banco de dados, geração de relatórios.
    • Gatilho HTTP (Webhook):
      • API RESTful: Transformar sua função em um endpoint HTTP que pode ser invocado por requisições HTTP (GET, POST, PUT, DELETE).
      • Níveis de Autorização: Configurar diferentes níveis de segurança (Function, Host, Admin, Anonymous) para controlar o acesso à sua função HTTP.
      • Integração com Webhooks: Receber webhooks de serviços como GitHub, Azure DevOps, Stripe, etc., para automatizar ações em resposta a eventos externos.

Desenvolver para armazenamento do Azure (15–20%)

Esta seção da certificação AZ-204 foca na implementação de soluções que interagem com os serviços de armazenamento do Azure, com ênfase em Azure Cosmos DB e Azure Blob Storage.

4. Desenvolver soluções que usam o Azure Cosmos DB

O Azure Cosmos DB é um banco de dados NoSQL totalmente gerenciado, multimodelo e distribuído globalmente.

4.1. Executar operações em contêineres e itens usando o SDK

  • Conceitos do Cosmos DB: Entender a hierarquia de recursos (conta, banco de dados, contêiner/coleção, item/documento) e os modelos de dados suportados (SQL API, MongoDB API, Cassandra API, Gremlin API, Table API). O foco principal para o AZ-204 é geralmente a SQL API.
  • Instalação e Configuração do SDK: Saber como adicionar o SDK apropriado ao seu projeto (ex: .NET SDK, Java SDK, Node.js SDK, Python SDK).
  • Operações CRUD (Create, Read, Update, Delete) em Itens:
    • Criação de Itens: Inserir novos documentos em um contêiner.
    • Leitura de Itens: Recuperar documentos por ID e chave de partição.
    • Atualização de Itens: Modificar documentos existentes.
    • Exclusão de Itens: Remover documentos.
  • Consultas (Queries):
    • Sintaxe SQL (para SQL API): Escrever consultas SQL para recuperar dados de contêineres.
    • Consultas Parametrizadas: Usar parâmetros para evitar injeção de SQL e melhorar o desempenho.
    • Paginação: Lidar com grandes conjuntos de resultados usando paginação.
  • Operações em Contêineres:
    • Criação e Exclusão de Contêineres: Saber como criar e remover contêineres programaticamente.
    • Throughput (RU/s): Entender o conceito de Request Units (RU/s) e como provisionar ou dimensionar o throughput para contêineres e bancos de dados.
  • Tratamento de Exceções: Implementar lógica para lidar com erros comuns, como conflitos e erros de throughput.

4.2. Definir o nível de consistência adequado para operações

A consistência é um conceito fundamental no Cosmos DB e impacta o desempenho e a durabilidade dos dados.

  • Níveis de Consistência: Entender os cinco níveis de consistência e seus trade-offs entre latência, throughput e durabilidade:
    • Forte (Strong): Maior consistência, mas maior latência e menor throughput. Garante que todas as leituras vejam a versão mais recente dos dados.
    • Consistência Limitada (Bounded Staleness): Consistência com um limite configurável de defasagem (tempo ou número de versões).
    • Sessão (Session): Consistência garantida para a sessão do cliente. É o padrão e o mais comum para a maioria das aplicações.
    • Prefixo Consistente (Consistent Prefix): Garante que as leituras nunca verão dados fora de ordem.
    • Final (Eventual): Menor consistência, mas maior latência e menor throughput. As leituras podem ver dados defasados.
  • Escolha do Nível de Consistência: Saber quando usar cada nível de consistência com base nos requisitos da aplicação.
  • Configuração: Entender como definir o nível de consistência no nível da conta e como substituí-lo para operações específicas usando o SDK.

4.3. Implementar notificações do feed de alterações

O Change Feed do Cosmos DB é um recurso poderoso para processamento de dados em tempo real.

  • Conceito de Change Feed: Entender que o Change Feed fornece um log persistente de todas as alterações feitas em um contêiner do Cosmos DB.
  • Casos de Uso:
    • Processamento de Dados em Tempo Real: Sincronização de dados, streaming de dados para outros serviços, atualização de caches.
    • Integração: Envio de eventos para Event Hubs, Service Bus ou Azure Functions.
    • Auditoria: Manter um registro de todas as modificações de dados.
  • Processador do Feed de Alterações (Change Feed Processor):

  • Imutabilidade do Blob (Immutable Storage):

  • Soft Delete para Blobs:
    • Conceito: Habilitar a exclusão suave para blobs, que retém os dados por um período configurável após a exclusão, permitindo a recuperação.
    • Casos de Uso: Proteção contra exclusão acidental.

7.1. Dados seguros de configuração de aplicativo usando a Configuração de Aplicativos ou o Azure Key Vault

  • Azure App Configuration:

  • Azure Key Vault:
    • Conceito: Entender o Key Vault como um serviço de nuvem para armazenar e gerenciar com segurança chaves criptográficas, segredos (como senhas, strings de conexão) e certificados.

  • Escolha entre App Configuration e Key Vault: Saber quando usar cada serviço. O Key Vault é para segredos de alto valor e chaves criptográficas, enquanto o App Configuration é para configurações de aplicação mais gerais (que podem incluir referências a segredos do Key Vault).

7.2. Desenvolver um código que use chaves, segredos e certificados armazenados no Azure Key Vault

  • Acesso ao Key Vault:

  • Operações com o SDK do Key Vault:
    • SecretsClient/SecretClient (SDKs modernos): Saber como usar os SDKs apropriados para:
      • Obter Segredos: Recuperar segredos por nome e versão.
      • Definir Segredos: Criar ou atualizar segredos.
      • Excluir Segredos: Remover segredos (com soft-delete).
    • Chaves e Certificados: Embora o foco do AZ-204 seja mais em segredos, ter uma compreensão básica do armazenamento e recuperação de chaves e certificados também é útil.
  • Best Practices:
    • Rotação de Segredos: Implementar mecanismos para rotacionar segredos regularmente.
    • Soft Delete e Purge Protection: Entender esses recursos para proteger contra exclusão acidental ou maliciosa.

7.3. Implementação de identidades gerenciadas para recursos do Azure

  • Conceito de Identidades Gerenciadas:
    • Autenticação sem Credenciais: Entender que identidades gerenciadas fornecem uma identidade para seus serviços Azure (App Service, Functions, VMs, etc.) no Microsoft Entra ID, eliminando a necessidade de gerenciar credenciais em seu código.
    • Tipos:
      • Atribuídas pelo Sistema: Criadas automaticamente com o recurso Azure e têm o mesmo ciclo de vida.
      • Atribuídas pelo Usuário: Criadas independentemente e podem ser atribuídas a múltiplos recursos.
  • Criação e Atribuição: Saber como habilitar identidades gerenciadas (tanto atribuídas pelo sistema quanto pelo usuário) para recursos Azure.
  • Atribuição de Funções (RBAC): Conceder permissões de Role-Based Access Control (RBAC) às identidades gerenciadas para que elas possam acessar outros serviços do Azure (ex: Azure Key Vault, Azure Storage, Azure SQL Database).
  • Utilização no Código:
    • Azure.Identity Library (para .NET/Python/Java): Usar a biblioteca Azure.Identity para obter automaticamente tokens de acesso para a identidade gerenciada, permitindo que o aplicativo se autentique em outros serviços do Azure.
    • Exemplo: Acesso a um blob no Azure Storage ou a um segredo no Key Vault usando a identidade gerenciada, sem strings de conexão ou chaves embutidas.

8.1. Monitorar e solucionar problemas de soluções usando o Application Insights

O Application Insights é um serviço de Gerenciamento de Desempenho de Aplicativos (APM) para desenvolvedores web profissionais. Ele monitora suas aplicações ao vivo, automaticamente detectando anomalias de desempenho.

8.1.1. Monitorar e analisar métricas, logs e rastreamentos

  • Rastreamentos (Traces) e Dependências:

8.1.2. Implementar alertas e testes Web do Application Insights

  • Alertas:
    • Criação de Regras de Alerta: Configurar regras de alerta baseadas em métricas (ex: taxa de CPU alta, tempo de resposta lento, taxa de falhas), logs (ex: contagem de exceções específicas excedendo um limite) ou na disponibilidade.
    • Grupos de Ação (Action Groups): Definir ações a serem tomadas quando um alerta é disparado (enviar e-mail, SMS, chamar um webhook, disparar um Azure Function, criar um ticket ITSM).
    • Severidade: Entender os diferentes níveis de severidade dos alertas.
  • Testes de Disponibilidade (Web Tests):
    • Testes de Ping de URL: Criar testes simples para verificar se um endpoint web está respondendo.
    • Testes de URL Padrão (Standard URL Tests): Testes mais avançados que podem validar conteúdo de resposta, cabeçalhos e SSL.
    • Testes de Várias Etapas (Multi-step Web Tests): Gravar e reproduzir sequências de requisições web para simular cenários de usuário mais complexos (ex: login, adicionar item ao carrinho).
    • Localizações de Teste: Configurar testes para serem executados de várias localizações geográficas para verificar o desempenho global.
    • Alertas de Disponibilidade: Configurar alertas com base nos resultados dos testes web (ex: se o teste falhar em N localizações).

8.1.3. Instrumentar um aplicativo ou serviço para usar o Application Insights

  • Adicionar SDK do Application Insights:
    • Bibliotecas de Telemetria: Saber como adicionar os pacotes SDK apropriados para sua tecnologia (ex: Microsoft.ApplicationInsights.AspNetCore para .NET Core, applicationinsights para Node.js, opencensus-ext-azure para Python).
    • Strings de Conexão/Chaves de Instrumentação: Configurar o SDK com a string de conexão ou chave de instrumentação do seu recurso do Application Insights.
  • Configuração Automática (Auto-instrumentation):
    • Azure Web Apps: Habilitar o Application Insights no nível do Serviço de Aplicativo sem modificações de código para runtimes suportados (ex: .NET, Node.js).
    • Azure Functions: Habilitar o Application Insights diretamente no Function App.
    • Contêineres/VMs: Usar a extensão Azure Monitor para VMs ou agentes OpenTelemetry/Application Insights para contêineres para coletar telemetria.

    • TrackEvent: Registrar eventos de negócio (ex: "ItemAdicionadoAoCarrinho").
    • TrackMetric: Enviar pontos de dados numéricos (ex: tempo de processamento de uma função interna).
    • TrackTrace: Enviar mensagens de log personalizadas.
    • TrackException: Registrar exceções capturadas.
    • TrackDependency: Rastrear chamadas para serviços externos ou bancos de dados que não são automaticamente detectados.
  • Contexto de Operação e Correlação: Entender como o Application Insights correlaciona automaticamente telemetria de diferentes componentes da sua aplicação (via ID de Operação), permitindo a visualização de transações completas.

9.1.4. Implementar políticas para APIs

As políticas do APIM são uma forma poderosa de alterar o comportamento das APIs através de XML.

  • Tipos de Políticas: Entender os diferentes escopos de política (global, produto, API, operação).
  • Políticas Comuns:
    • Controle de Acesso: ip-filter (filtrar por IP), check-header (verificar cabeçalhos).
    • Limitação de Taxa (Rate Limiting): rate-limit-by-key (limitar o número de chamadas por chave de assinatura ou usuário).

    • Transformação de Dados: set-header, set-body, rewrite-uri (modificar cabeçalhos, corpo da requisição/resposta, reescrever URLs).

  • Encadeamento de Políticas: Entender como as políticas são processadas em uma sequência (entrada, saída, erro).

  • Conceito de Azure Event Hubs: Entender o Event Hubs como um serviço de ingestão de streaming de dados em larga escala, capaz de receber milhões de eventos por segundo. Ideal para cenários de Big Data, IoT e telemetria.

9.3. Desenvolver soluções baseadas em mensagem

Este tópico se concentra nos serviços de mensagens para comunicação assíncrona e desacoplamento de componentes.

9.3.1. Implementar soluções que usam o Barramento de Serviços Azure

  • Conceito de Azure Service Bus: Entender o Service Bus como um broker de mensagens corporativo gerenciado que suporta filas e tópicos. Ideal para integração de aplicações empresariais e comunicação assíncrona transacional.
  • Filas (Queues):
    • Casos de Uso: Comunicação ponto a ponto, balanceamento de carga, garantia de entrega (at-least-once).
    • Envio de Mensagens: Enviar mensagens para uma fila usando os SDKs.
    • Recebimento de Mensagens: Receber e processar mensagens de uma fila (peek-lock, receive-and-delete).
    • Dead-Letter Queue (DLQ): Entender o DLQ para mensagens que não puderam ser processadas.
  • Tópicos e Assinaturas (Topics and Subscriptions):
    • Casos de Uso: Publicar/assinar (publish/subscribe) para cenários de comunicação um-para-muitos.
    • Regras de Filtro: Usar filtros em assinaturas para que os consumidores recebam apenas mensagens específicas de um tópico.
  • Recursos Avançados:
    • Sessões de Mensagem: Entender sessões para processamento de mensagens ordenadas ou agrupadas.
    • Agendamento de Mensagens: Enviar mensagens para serem processadas em um horário futuro.
    • Transações: Agrupar operações de envio/recebimento de mensagens em uma única transação.
  • Autenticação: Usar Identidades Gerenciadas ou Assinaturas de Acesso Compartilhado (SAS) para autenticar no Service Bus.

9.3.2. Implementar soluções que usam filas de Armazenamento de Fila Azure

  • Conceito de Azure Queue Storage: Entender o Queue Storage como um serviço de fila de mensagens simples e econômico, construído sobre o Azure Storage. Ideal para cenários de desacoplamento de tarefas e processamento assíncrono básico.
  • Filas e Mensagens:
    • Casos de Uso: Processamento de tarefas em segundo plano, comunicação simples entre componentes.
    • Adicionar Mensagens: Adicionar mensagens a uma fila.
    • Recuperar Mensagens: Obter mensagens da fila (com timeout de visibilidade).
    • Excluir Mensagens: Remover mensagens após o processamento bem-sucedido.
    • Contagem de Dequeue: Monitorar quantas vezes uma mensagem foi recuperada.
  • Diferenças do Service Bus: Entender as principais diferenças entre Queue Storage e Service Bus (escala, recursos, garantias de mensagens, complexidade), e quando usar um em vez do outro. Queue Storage é mais simples e econômico para filas básicas, enquanto Service Bus oferece recursos de mensagens corporativas mais avançados.
